# Referee 4 — Reproducibility review (Royal Society Open Science)

**Recommendation:** Major revision

**1-sentence summary:** The core 6,630-run elicitation dataset is solid and fully reproducible (tests pass, the comparison/rollup CSVs and 32 of 34 paper tables regenerate byte-for-byte from committed artifacts, and the headline count/parse-rate/benchmark numbers verify against raw `runs.jsonl`), but the manuscript ships with a rendered internal contradiction in its headline policy table, an inconsistent cost total, several stale rankings that disagree with the committed tables, an untraceable "58 failures" claim, and none of the data-accessibility / funding / competing-interests / ethics statements RSOS requires.

I verified everything below against committed artifacts and local re-runs only. I made no API calls and spent no money. Test suite: `71 passed in 2.09s` (`.venv/bin/python -m pytest -q`).

---

## Major comments

### 1. The headline optimal-top-tax table (Table 4) and its robustness table (Table A13) were built in PolicyEngine *fallback* mode (a = 1.500) and directly contradict the prose that surrounds them (a = 1.470). The contradiction is baked into the rendered PDF/HTML.

`build_tables.py` shells out to a PolicyEngine-US venv at a hard-coded path (`/Users/maxghenis/PolicyEngine/policyengine-us/.venv/bin/python`, [build_tables.py L37-L44](../build_tables.py#L37-L44)). When that venv is absent it prints a warning and substitutes fallback constants: `estimate_policyengine_top_tail_pareto_parameter()` returns `a = TOP_RATE_PARETO_A = 1.5` with `threshold = NaN` ([build_tables.py L1425-L1437](../build_tables.py#L1425-L1437)), and `estimate_policyengine_flat_tax_demogrant_frontier()` returns `{"rows": []}` ([build_tables.py L1498-L1500](../build_tables.py#L1498-L1500)).

The committed tables are the *fallback* build, which I proved two ways:
- The committed [`toy-top-rate-labor-tax.md` L1](../tables/toy-top-rate-labor-tax.md#L1) note literally reads "…using log utility (gamma = 1) and **fallback Pareto parameter a = 1.500**." The committed [`top-rate-robustness.md` L3](../tables/top-rate-robustness.md#L3) has a duplicate "Baseline (a=1.500)" and "a=1.5" column that are identical cell-for-cell — a tell-tale that the microdata estimate equalled the 1.5 fallback.
- I backed up `paper/tables/`, re-ran `build_tables.py` with `POLICYENGINE_US_REPO=/nonexistent` (forcing the fallback, no microsim, no cost), and **all 34 tables regenerated byte-for-byte identically**. The committed `toy-top-rate-labor-tax.md` *is* the a=1.500 fallback output.

But the manuscript prose describes a real microdata calibration: "In the current build, that estimate is `a = 1.470`" with "threshold `$659,618`… tail mean `$2,062,980`" and formula `τ* = 0.405 / (0.405 + 1.470 e)` ([paper.qmd L231-L240](../paper.qmd#L231-L240)), and quotes top-rate medians "`33.5%` for GPT-5.4 nano to `44.3%` for Gemini 3.1 Pro, with GPT-5.4 at `39.6%`" ([paper.qmd L247](../paper.qmd#L247)). Those three numbers match an a=1.470 build; the committed table shows **43.2% / 38.8% / 32.8%** for the same three models (a=1.500). I confirmed the rendered `paper.html` contains **both** "a = 1.470" (2×) and "fallback Pareto parameter a = 1.500" (1×), and both the prose rates (44.3/39.6/33.5) and the table rates (43.2/38.8/32.8). A reader sees a Table 4 whose numbers and note contradict the prose two paragraphs above.

Table A12 (`flat-tax-demogrant-appendix.md`) compounds this: it contains real microsim dollar values (e.g. `$21,320` median, `$90,500` P90), so it came from an *earlier* build when the venv was present. Because the fallback path skips the write (`if flat_tax_frontier:` at [build_tables.py L297](../build_tables.py#L297)), the last (fallback) build left that stale table in place. So the committed table set is not from a single coherent build: A12 is a=1.470-era, Table 4 / A13 are a=1.500 fallback, and the prose is a=1.470.

**Consequence for reproducibility:** an external reviewer (the intended audience of a "public repo") does not have `/Users/maxghenis/PolicyEngine/policyengine-us` and will reproduce exactly the committed a=1.500 tables, which contradict the prose. A reviewer who *does* install policyengine-us will regenerate a=1.470 tables that *differ* from the committed ones. Table 4/A13 are wrong relative to something no matter what. This is the single highest-priority fix.

### 2. Cost accounting is internally inconsistent: April totals exclude the clarify probes, July totals include them, and the "$60.89 across 6,630 successful runs" headline misattributes cost to runs that are not in the panel.

I reconstructed every reported cost from the committed summaries ([paper.qmd L162-L185](../paper.qmd#L162-L185)):
- Each **April** per-model figure equals that model's `elasticity-model-rollup.csv` `total_cost_usd` (elasticities panel only). April total `$23.15` = the 11-model rollup sum. Verified to the cent.
- Each **July** per-model figure equals `elasticities + armington-clarify + ies-clarify` summary costs. E.g. Claude Fable 5: `16.7977 + 0.5502 + 0.6528 = 18.001`; GPT-5.5: `5.4182 + 0.1493 + 0.2002 = 5.768`; all six match to the cent. July total `$37.74` = the six all-in sums.

So the two halves use different conventions. Applied consistently, the grand total is either **$58.33** (elasticities only: 23.15 + 35.18) or **$62.76** (all-in: 25.02 + 37.74) — not the reported **$60.89** (= 23.15 + 37.74, mixed). Moreover the two clarify probes are **510 runs** (255 armington + 255 ies) that are *not* part of the "6,630 successful runs" ([paper.qmd L158](../paper.qmd#L158); 6,630 = 17×26×15 is the elasticities panel). Attributing "$60.89 across 6,630 successful runs" is therefore incorrect on both counts. Separately, `claude-opus-4.7`'s armington-clarify cost cell is empty (the `$0.0000` per-quantity-merge artifact documented at [results/README.md L47-L53](../../results/README.md#L47-L53)), so any all-in April accounting silently undercounts it.

The root cause is that the cost figures are hand-transcribed prose, not generated by `build_tables.py`. They should be emitted from the artifacts so they cannot drift.

### 4. Table notes hard-code "11 models" even though every table now has 17 model rows.

`build_tables.py` writes fixed strings "9 quantities x 11 models x 15 runs", "6 quantities x 11 models", "3 quantities x 11 models" ([build_tables.py L212-L213](../build_tables.py#L212), [L230](../build_tables.py#L230), [L238](../build_tables.py#L238)), which render into [model-overview.md L1](../tables/model-overview.md#L1) etc. I confirmed the corresponding tables carry 17 model rows. These counts should be computed from the data, not literalized.

### 5. RSOS-required end-matter is entirely absent, and there is no persistent archive or license.

`paper.qmd` contains no **Data accessibility**, **Code availability**, **Competing interests**, **Funding**, **Authors' contributions**, or **Ethics** statement (grep returns only the two prose uses of "reproducible"). RSOS mandates the first five. There is also **no `LICENSE`, no `CITATION.cff`, no `.zenodo.json`**, and no DOI/persistent-archive reference anywhere in the manuscript — only an implied GitHub URL. RSOS requires data and code to be deposited in a citable archive (Zenodo/Dryad/Figshare) with a DOI at acceptance; a GitHub URL alone is insufficient, and with no license the repository is technically all-rights-reserved.

### 6. The "58 re-elicited infrastructure failures" are asserted in prose but have no traceable artifact in the repository.

[paper.qmd L160](../paper.qmd#L160) gives a detailed breakdown (empty `gpt-5.5` responses, missing `grok-4.3` tool calls, three `claude-sonnet-5` >32k-token responses, one schema-compilation error, one safety-classifier false positive). But `rerun_failed_runs.py` re-elicits failed slots **in place** ([rerun_failed_runs.py L1-L12](../../scripts/rerun_failed_runs.py#L1-L12)), and the committed data shows the *post-repair* state only: I checked all 17 `*-elasticities-batch15/runs.jsonl` and found 6,630 records with `parsed_ok == True` for 100.0%, `error` empty for every record, and `requests.jsonl` containing only successful requests (e.g. `gpt-5.5` has 118 request rows and no failure field). The number "58" appears nowhere in the tracked source, and there is no failure manifest, log, or dated diff. The claim is plausible and the mechanism is documented, but as committed it is neither verifiable nor reproducible. A committed failure ledger (model, quantity, run index, original error, replacement request id) is needed.

## Verified (for the record — these are genuinely reproducible)

- **Tests pass:** `71 passed in 2.09s`.
- **6,630 / 100% parse / v4 / tool-free** all verify against raw `runs.jsonl`: exactly 6,630 records across 17 `*-elasticities-batch15` dirs, `parsed_ok == True` for 100.00%, zero non-empty `error` fields, `prompt_version = v4` and `tool_regime = none` throughout, 390 runs per model (26×15), 18 quantile-repairs tracked.
- **Derived CSVs regenerate byte-for-byte:** `scripts/build_comparison_artifacts.py` reproduces `elasticity-all-model-comparison.csv` (442 = 17×26 rows) and `elasticity-model-rollup.csv` identically from the committed `summary.csv` files; registry has 26 quantities.
- **32 of 34 tables regenerate byte-for-byte** from committed results in fallback mode (only the two PolicyEngine-backed tables carry the provenance problem in comment 1).
- **Benchmark numbers verify:** in-range counts (17/17 Frisch; 16/17 cap-gains, wage, single-mother; 14/17 income; 15/17 ETI) and the sign-sensitive and ETI centers cited in the text (gemini-3.5-flash +0.010/+0.011, gpt-5.5 −0.035, haiku 0.507, nano 0.552, sonnet-4.6 0.500, sonnet-5 0.473) all match `elasticity-all-model-comparison.csv`.
- **Quarto rendering is clean:** the manuscript uses `{{< include >}}` shortcodes and **zero** inline `{python}` expressions, so none of the string-in-math-span / escaped-table / double-brace failure modes apply; `paper.html` contains no `\.` inside math spans, no `&lt;table&gt;`/`&lt;tr&gt;`, and no `{{` inside math spans.
